# KOSHA 엔터프라이즈 격상 계획 V0.1 (KO)

## 1. 문제 정의

현재 시스템은 계산 로직 자체는 강하지만, 의사결정 화면에서 규제 근거가 즉시 보이지 않으면 일반 계산기로 인식될 수 있습니다.

EPC 엔터프라이즈 고객의 구매 기준은 주로 아래 3가지입니다.
- 계산 신뢰성,
- 표준/법령 추적성,
- 감사 제출 가능성.

## 2. 전략 원칙

디자인 원칙: `Industrial Modernism`.

핵심 규칙:
- 정보 밀도 극대화,
- 화려함보다 신뢰 신호 우선,
- 실무 작업에 맞는 고정형/컴팩트 레이아웃 유지.

목표 레이아웃:
- 좌측 25%: 입력/조건,
- 중앙 50%: 계산 결과/추세,
- 우측 25%: 검증/KOSHA GUIDE/법령/준수상태.

## 3. KOSHA API 차별화

승인된 두 API를 결합합니다.
- KOSHA GUIDE: "어떻게 설계/검사할 것인가"
- 법령 스마트검색: "왜 의무이며 미준수 리스크가 무엇인가"

결과 표현 전환:
- 이전: 수치 결과 중심.
- 이후: 수치 + 기술지침 근거 + 법령 근거 + 준수 요약.

## 4. 실행 단계

### Phase 1 (현재 스프린트)
- 3-Pane 고밀도 레이아웃 구현.
- 기술지침/법령 API 연동.
- 우측 패널에 규제 컨텍스트와 상태 배지 표시.

### Phase 2 (다음 스프린트)
- 공종별/리스크별 매핑 규칙 고도화.
- 캐시 및 장애 fallback 강화.
- 감사 제출용 리포트 템플릿 확장.

### Phase 3
- 검색/매핑 품질 KPI 대시보드 추가.
- 계정/역할 기반 정책 프로파일(운영/검사/관리) 도입.

## 5. 성공 지표

- `기술지침 매핑 커버리지`: 계산 1건당 2건 이상(가능한 경우).
- `법령 매핑 정밀도`: 전문가 검토 기준 80% 이상 유효 조항.
- `감사 대응`: 수식+표준+법령 근거를 한 번에 제출 가능.
- `응답속도`: 캐시 적용 상태에서 실사용 지연 최소화.

## 6. 기술 가드레일

- `frontend/lib/kosha-regulatory.ts`는 오케스트레이션 역할만 유지.
- 파싱/쿼리/매핑/준수평가 로직은 `frontend/lib/kosha/*`로 모듈 분리.
- API 키/엔드포인트/응답 장애 시 fallback 동작 보장.
- 키는 `.env`로만 관리하고 코드 하드코딩 금지.

## 7. 즉시 실행 항목

1. 주요 리팩터링마다 스냅샷 백업 생성.
2. 규제 로직 모듈 경계와 테스트 가능성 유지.
3. 우측 패널은 고밀도/근거중심/감사지향으로 유지.
4. 신뢰/준수 속도를 높이지 못하는 장식성 UI 작업은 후순위.
